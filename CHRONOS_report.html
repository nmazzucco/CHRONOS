Executive summary

The CHRONOS script implements a reproducible workflow for classifying obsidian assemblages by technological production involvement and modelling potential circulation pathways across the Central Mediterranean during two Neolithic chronological intervals: 6100–5500 cal BCE and 5500–4900 cal BCE.

Primary objective

Estimate site roles in obsidian production and circulation by combining technological composition, production-score ranking, geographic distance, rank-based interaction rules, and source attribution.

Main analytical units

Sites with coordinates, obsidian technological counts, total artefact counts, and source-presence information for Sardegna / Monte Arci, Lipari, and Palmarola.

Main outputs

Ranked site tables, Brainerd–Robinson diagnostic matrices, PCoA plots, rule–distance network sweeps, centrality tables, source-confirmed subnetworks, and publication-ready maps.

Important: this HTML report documents the script and its expected workflow. It does not contain numerical results because the raw Excel datasets and shapefile were not executed here.

Scope and research aims

The script is designed to evaluate how obsidian production, transformation, and circulation may have been organised across Central Mediterranean Neolithic communities. It focuses on two chronological windows and compares alternative interaction models based on distance thresholds and production-rank rules.

The archaeological problem addressed by the workflow is not simply whether sites were connected, but under which assumptions a coherent circulation structure emerges. The script therefore tests multiple rule sets, evaluates the distance threshold needed to connect the system, and extracts source-confirmed subnetworks for Monte Arci / Sardegna, Lipari, and Palmarola.

Analytical workflow

Read and clean input data. The script reads XLSX/XLSM or CSV files, normalises column names with make.names(), standardises site names, converts coordinates and counts to numeric values, and removes records without valid site names, coordinates, or total counts.
Calculate production-contribution percentages. For each technological class, the script calculates each site’s percentage contribution to the total quantity of that class across the full dataset.
Assign production ranks. A production score is calculated as the row sum of contribution percentages. Sites in the upper 10% are assigned Rank 3, sites between the 80th and 90th percentiles Rank 2, and all remaining sites Rank 1.
Add raw-material source nodes. Monte Arci, Lipari, and Palmarola are manually added as Rank 4 nodes with geographic coordinates and source indicators.
Build rule–distance networks. Directed networks are created for several rank-based rule sets and distance thresholds from 100 to 650 km at 50 km increments.
Evaluate network structure. The script exports global metrics, centrality measures, rule-comparison plots, and statistical comparisons among rule sets.
Extract source-confirmed subnetworks. For each source, only the source node and sites with confirmed presence of that source are retained. Rule A is attempted first, with Rule B as a fallback when the source-confirmed graph is uninformative.
Generate maps and graph visualisations. The script exports network maps, graph-only layouts, PCoA plots, source-confirmed maps, and a numbered black-and-white site location map.

Production-score ranking

The ranking procedure avoids unsupervised clustering and instead uses an explicit, reproducible production-involvement score. This score is intended to estimate the quantitative contribution of each site to obsidian production and transformation.

Production_score = blade.PN + flake.PN + core.PN + by.products.waste.PN
RankInterpretationOperational definition
Rank 1Low production involvement / consumer-oriented assemblageBelow the 80th percentile of production score.
Rank 2Intermediate production involvementBetween the 80th and 90th percentiles.
Rank 3High production involvement / producer-oriented assemblageAt or above the 90th percentile.
Rank 4Raw-material sourceManually added nodes: Monte Arci, Lipari, Palmarola.

Brainerd–Robinson-style dissimilarity and PCoA are retained as diagnostic tools to evaluate whether the production-score ranks correspond to meaningful multivariate differences in technological space. The script also applies a PERMANOVA-style test using vegan::adonis2().

Network modelling

Networks are built from rank-based allowed connections and maximum centre-to-centre geographic distance. Distances are calculated after transforming site coordinates from WGS84 to ETRS89 / LAEA Europe (EPSG:3035), which is appropriate for broad European distance calculations.

Rule setAllowed directed connectionsInterpretation
Rule A4-3, 3-2, 3-1, 2-1Parsimonious production-distribution model excluding direct source-to-intermediate links.
Rule B4-3, 4-2, 3-2, 3-1, 2-1Less restrictive model allowing direct links from sources to intermediate sites.
Rule C4-3, 3-2, 2-1Strictly hierarchical model that excludes direct producer-to-consumer links.

For each rule set, the script evaluates distance thresholds from 100 km to 650 km in 50 km steps. For every model, it exports edge lists, maps, global metrics, and centrality values.

Network metrics

The script summarises each model using number of nodes, edges, connected components, isolates, largest component size, and whether all sites form one connected component. It also calculates degree, betweenness, closeness, harmonic centrality, and eigenvector centrality.

Source-confirmed networks

Source-confirmed networks are extracted for three obsidian sources:

  • Monte Arci / Sardegna
  • Lipari
  • Palmarola

The source-confirmed filter retains only the selected source node and archaeological sites where the selected source column equals 1. Sites with 0 or NA for that source are excluded from the confirmed-source subgraph. Other raw-material source nodes are also excluded.

The script first attempts source filtering on the selected Rule A backbone. If the source-confirmed graph is empty, has no edges, lacks the source node, or leaves the source node isolated, it falls back to Rule B and records the fallback reason in the output log.

Recommended reproducibility additions

  • Add set.seed(123) immediately before force-directed graph layouts as well as at the top of the script, because layouts can vary across sessions.
  • Include an renv.lock file or equivalent package-version record.
  • Add a data dictionary describing each input column and the meaning of source values 1, 0, and NA.
  • Export or document the coordinate reference system of the input site coordinates and shapefile.
  • Consider adding a short script header with author, ORCID, repository URL, version, date, and licence.
