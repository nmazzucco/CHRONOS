1. Analytical aim and report scope

The CHRONOS workflow models the production and circulation of obsidian in the Central Mediterranean by combining technological assemblage data, production-rank classification, geographical distance thresholds, rank-based connection rules and source-confirmed subnetwork extraction.

Production ranking

Sites are ranked according to their contribution to the total assemblage of blades, flakes, cores and by-products/waste. The resulting ranks are used as archaeological roles in network construction.

Network sensitivity

Three rule sets are evaluated across thresholds from 100 to 650 km. The report records maps, metrics and statistical comparisons for each rule-distance combination.

Source-confirmed networks

Separate networks are extracted for Monte Arci/Sardegna, Lipari and Palmarola using confirmed source attribution, with Rule A preferred and Rule B used as fallback when needed.

How to use this HTML on GitHub. Place CHRONOS_report.html in the repository root, together with the OUTPUTS/ folder produced by the R script. The figure cards below use relative paths and will display automatically when the PNG files are present.

3. Step-by-step workflow

Read and clean data

The script reads Excel or CSV files, normalises column names with make.names(), standardises the site-name column, converts coordinates and technological counts to numeric values, and removes records without valid site names, coordinates or total counts.

Key outputs: cleaned tables are not exported directly at this stage, but they are embedded in later ranked-site outputs.

Calculate production-contribution percentages

For each technological class, each site is expressed as a percentage contribution to the total amount of that class across the full dataset. These values are not row-rescaled; they preserve the magnitude of production involvement.

Key output: production_contribution_PN_all_sites.csv.

Calculate Brainerd-Robinson diagnostic similarity

A Brainerd-Robinson-style similarity matrix is calculated on the production-contribution values. In this version of the workflow the matrix is diagnostic, not classificatory.

Key outputs: obs_br_PN_all_sites.csv, adonis_BR_dissimilarity_by_rank_all_sites.csv, pcoa_BR_by_production_rank_all_sites.csv, PCoA_BR_by_production_rank_all_sites.png.

Assign production ranks

The production score is calculated as the row sum of the four production-contribution variables. Rank 3 corresponds to the upper 10% of the production-score distribution, Rank 2 to the 80th-90th percentile, and Rank 1 to all remaining sites.

Key outputs: production_score_all_sites.csv, production_score_rank_thresholds_all_sites.csv, site_production_score_ranking_all_sites.csv, ranked_sites_all_sites.csv, rank_summary_all_sites.csv.

Attach full-dataset ranks to chronological subsets

Site names are standardised by removing non-breaking spaces, reducing repeated whitespace, transliterating to ASCII and converting to lower case. Full-dataset ranks are then joined to the two chronological datasets.

Key outputs: ranked_sites_6100_5500.csv, ranked_sites_5500_4900.csv, plus unmatched-site QC tables.

Add raw-material source nodes

Monte Arci, Lipari and Palmarola are manually inserted as Rank 4 nodes with coordinates and source-presence markers. These nodes make the network explicitly source-aware.

Key effect: all chronological network models include archaeological sites and the three source nodes.

Build rule-distance network models

Directed edges are created when two sites fall within a specified centre-to-centre distance and their rank transition is allowed by the selected rule set. Distances are calculated in ETRS89 / LAEA Europe (EPSG:3035).

Rule A: 4-3, 3-2, 3-1, 2-1. Rule B: 4-3, 4-2, 3-2, 3-1, 2-1. Rule C: 4-3, 3-2, 2-1.

Key outputs: 36 edge lists and 36 map PNGs per chronological interval, plus global metrics and centrality tables.

Compare rule sets statistically

For each interval, the script compares network metrics across rule sets using Friedman tests with distance threshold as a blocking factor, followed by pairwise Wilcoxon signed-rank tests.

Key outputs: friedman_rule_comparison_*.csv and pairwise_rule_comparison_*.csv.

Select connected and parsimonious backbones

The script identifies the first connected network for each rule set and records selected backbones for downstream analysis. Rule A is used as the preferred parsimonious backbone for source-filtered networks, with Rule B available as fallback.

Key outputs: first_connected_network_by_rule_*.csv, selected_backbone_network_*.csv, selected_rule_a_source_filter_backbone_*.csv.

Build source-confirmed networks

For each obsidian source, the script retains the selected source node and only archaeological sites with confirmed presence of that source. Rule A is attempted first; Rule B is used only if the Rule A graph is empty, has no edges, lacks the source node or isolates the source node.

Key outputs: source-confirmed edge/vertex lists, source maps, source graph-only figures, rule-selection logs and source metrics.

Create the publication location map

The final map plots all sites with numbered labels and exports a lookup table linking map IDs to site names and coordinates.

Key outputs: all_sites_numbered_black_white_map.png and all_sites_numbered_map_lookup.csv.

6. Reproducibility notes for GitHub

Package note. The script calls vegan::vegdist() and vegan::adonis2() but does not explicitly load vegan in the package block. Because the functions are namespace-qualified, the code can work if vegan is installed; nevertheless, add library(vegan) or list vegan in the repository dependencies.

Recommended repository files

Add a README.md, renv.lock or package list, LICENSE, CITATION.cff, and a short note on data availability.

Recommended data policy

If raw archaeological data cannot be fully public, include a synthetic example dataset and document access restrictions for the real files.

Recommended figure policy

Keep the generated PNG files under OUTPUTS/ so the HTML can display them directly on GitHub Pages or in a browser.

7. Publication checklist

  • Confirm that unmatched-site tables are empty or explain any unresolved unmatched records.
  • Check consistency between comments and active backbone-selection code before submission.
  • Report the selected connected threshold for each chronological interval using first_connected_network_by_rule_*.csv.
